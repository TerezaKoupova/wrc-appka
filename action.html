PostgreSQL + Auth + Storage. RLS na každé tabulce. Dva projekty — app-dev (lokál) a app-prod (Vercel).

GitHub
Repo, issues, PR, Actions. Source of truth — všechno žije v gitu včetně SQL migrací a PRD.
→ Verzování + issue tracker
Vercel
Hosting pro Next.js. Auto-deploy z mainu, preview URL na každém PR (= staging zdarma).
→ Živá *.vercel.app URL

Claude píše kód → push do GitHubu → Vercel automaticky deployne → appka volá Supabase. Next.js je středobod.
Volitelně: Gemini API (AI feature, free tier) · Brevo (emaily, 300/den zdarma) — sahej po nich, až je feature potřebovat bude.

Blok 4

Review scaffoldu — než deployne

Druhý pár očí přes scaffolded kód. Najdeš věci, které sám nevidíš — než to půjde na produkci.

/hack-review
Druhá AI projde celou scaffolded appku. Hledá blockery, bezpečnost, soulad s PRD, mrtvý kód.
→ Review report (max 5 bodů: blockery / warningy / nitpicky)
Co s nálezy
Blocker = oprav teď (RLS chybí, .env v gitu, klíče v kódu). Warning = oprav před první feature. Nitpick = ignoruj nebo si zaloguj jako issue.
→ Vrať findings agentovi: „oprav blockery"
Proč zrovna teď
Scaffold je největší jednorázová generace v celém workshopu. Po něm se kód jen vrství. Lepší chytit problém tady, než ho odhalit po 3 features.
→ Čistý základ pro deploy + iterace
Výstup: Review report v terminálu, opravené blockery, čistý kód připravený k deployi.

Proč ne push na main?

Vaši appku už někdo používá.

Push rovnou do main = deploy rovnou do produkce. Každá chyba je live. Potřebujeme bezpečnostní síť.

BEZ SDLC

• Push na main → rovnou v produkci
• Uživatel vidí bug dřív než vy
• Hotfix → další bug → panika
• SQL z hlavy, klíče v kódu
• "Mně to funguje lokálně"

S SDLC

• Branch → implementace → PR
• Preview URL — otestuješ před mergem
• Code review — druhý pár očí / review agentem
• Migrace v gitu — historie DB změn
• .env.example — kolega ví co nastavit
Tohle není byrokracie. Je to hygiena vibecodera. Jako mytí rukou — děláš to vždycky, ne jen když se někdo dívá.
Od teď: nikdy push na main. Vždycky branch → PR → preview → merge.

/hack-test

Testy — pojistka proti regresi

AI generuje kód rychle. Ale taky rychle rozbije to, co už funguje.

PROČ TESTY U VIBE CODINGU

• AI při každé změně přepíše kus kódu
• Nevidíš co se rozbilo — nemáš mentální model
• Ručně proklikáváš celou appku po každém promptu
• Testy = automatický "proklik" za 2 sekundy
/hack-test
Agent se podívá do PRD a kódu, navrhne co testovat a napíše 5–8 testů.
→ vitest.config.ts + testy + npm test zelený
Co se testuje
• Utility funkce (formátování, validace)
• Komponenty (renderování, interakce)
• Mockovaný Supabase klient
Výstup: npm test běží zeleně. Máš základní safety net pro refaktoring.
/hack-ci

CI pipeline — automatická kontrola

Rozbitý kód se nepodaří mergnout. Zjistíš problémy dřív než na Vercelu.

PROČ CI U VIBE CODINGU

• AI generuje kód co "vypadá dobře" ale necompiluje
• Bez CI zjistíš problém až na Vercelu (nebo v produkci)
• CI donutí AI dodržovat strukturu a pravidla
• Rozbitý kód se nepodaří mergnout — red check na PR
/hack-ci
Agent vytvoří pipeline: lint, typecheck, testy, build. Běží automaticky na každém push a PR.
→ .github/workflows/ci.yml + zelený ✓ na PR
Když CI spadne: 1) Řekni Claudovi ať se podívá sám — dej mu link na spadlou akci. 2) Nebo zkopíruj chyby a copypastni mu je.
Výstup: Zelený check ✓ na každém PR. Broken build se nikdy nemergne.
Blok 7

Další feature — ale rovnou s reviewerem

Stejný feature loop, jen místo jednoho agenta máš tým. Lead řídí, Builder kóduje, Critic reviewuje.

TÝM TŘÍ ROLÍ

L
Lead — orchestrátor. Rozdělí práci, deleguje, hlasitě rozhoduje.
B
Builder — implementuje (Task subagent).
C
Critic — reviewuje diff (Task subagent).
Max 2 kola Build → Critique → Build → Critique. Pak eskalace na tebe — nekonečné smyčky odpadají.
/hack-feature-pro
Vybereš issue (klidně větší než obvykle). Lead rozdělí na úkoly, Builder kóduje, Critic ti reviewne diff dřív než ho vidíš ty. Iterace je vidět.
→ Feature branch + PR s vyšší kvalitou (review proběhl už uvnitř)
Kdy to použít
• Větší feature, kde čekáš nuance v UX nebo edge cases
• Chceš vidět "tým v akci" místo jednoho monologu
• Když by /hack-feature + /hack-review znamenaly dvě ruční kola
Výstup: PR, který už prošel inline review. Vidíš tři pojmenované role pracovat spolu — to je celá pointa.
Blok 8

Postavíme si vlastního agenta

Agent není žádná magie. Je to jen naučený kolega — markdown soubor s rolí, který si bereš s sebou do dalších projektů.

CO POSTAVÍME — prd-critic

• Subagent, co projde tvoje PRD a najde díry
• Píšeš ho ty sám — agent diktuje strukturu po blocích
• Žije v .claude/agents/prd-critic.md
• Otestuješ ho na vlastním PRD ze začátku workshopu
• Odneseš si ho — kopíruj do dalšího projektu, funguje stejně
Tři primitiva Claude Code: Command (zkratka pro flow) · Agent (přenosná role) · Team (Lead + subagenti). Ne ladder, ne hierarchie — tři různé nástroje.
/hack-agent
Reverse-engineeruje /hack-prd, ukáže že agent = markdown + role. Pak tě provede stavbou prd-critic blok po bloku — frontmatter, description (= trigger), instrukce.
→ .claude/agents/prd-critic.md, který sis napsal sám
Další nápady na subagenty
• commit-msg — z diffu vyrobí conventional commit
• seed-data — vygeneruje realistická data do DB
• meeting-notes — z poznámek udělá strukturovaný zápis
Výstup: Vlastnoručně napsaný subagent v repu. Už víš, jak si vyrobit dalšího — kdykoli, v jakémkoli projektu.
